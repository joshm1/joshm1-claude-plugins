# Branch Triage

Generate an interactive, mobile-friendly branch triage dashboard from live git/GitHub data. The dashboard lets the user review, prioritize, and take action on every unmerged branch — not just delete/keep, but a full triage workflow.

## Triage Actions

Each branch can be assigned one of these actions. The page should default to a sensible action based on the data (merged PR → delete, open PR → keep, etc.), but the user can override any of them:

| Action | When to use | What it means |
|--------|-------------|---------------|
| **Delete** | PR merged or closed, branch is stale cruft | Generate `git branch -D` / `git push origin --delete` commands |
| **Keep** | Open PR, active work, release automation | Leave it alone — it's doing its job |
| **Next Up** | Valuable unfinished work to tackle soon | Add to a prioritized work queue with drag-to-reorder |
| **Rebase & PR** | Good work that drifted from main, never PR'd | Flag for rebase onto main and PR creation |
| **Cherry-pick** | Branch is stale but has a few good commits | Flag individual commits worth salvaging before deleting |
| **Park** | Not ready to delete, not ready to work on | Explicitly deferred — out of sight but not forgotten |

## Bundled Resources

```
branch-triage/
├── SKILL.md
├── template.html              # Dashboard template (paste-in UI + embedded data support)
└── scripts/
    ├── gather_branch_data.py  # Gathers git/gh data, outputs JSON or AI prompt
    └── generate_html.py       # Injects JSON into template → final HTML
```

**`gather_branch_data.py`** — Run from any git repo. Gathers branch metadata, PR status (via `gh`), diff stats, commit lists, and safety verdicts (squash-commit verification). Uses only stdlib + subprocess (no pip dependencies).
- `--json` — Output raw JSON data (for the pipeline below)
- `--copy` — Copy output to clipboard
- `-o FILE` — Write output to file instead of stdout
- Default (no `--json`) — Output an AI prompt for the offline/phone workflow

**`generate_html.py`** — Takes a JSON data file and injects it into `template.html` to produce the final dashboard HTML. No manual HTML editing needed.

**`template.html`** — Supports two modes: (1) embedded data mode (when generated by `generate_html.py`) auto-loads and renders immediately, (2) paste-in mode for the offline/phone workflow.

```bash

```

When running the full skill (with terminal access), Claude should:
1. Run `gather_branch_data.py --json -o branch_data.json`
2. Read the JSON, analyze each branch, and fill in the `action` and `recommendation` fields
3. Write the completed JSON back to `branch_data.json`
4. Run `generate_html.py branch_data.json -o branch_triage.html`
5. Serve the HTML file

To update the dashboard later, just re-run steps 1-4. The template stays the same — only the data changes.

## Data Gathering

Collect branch metadata, PR status, and safety verification. Use **parallel subagents** to check branches concurrently — don't process them sequentially.

### Step 1: Get branch lists and PR status

Run these git commands (all in parallel) to build the initial dataset:

```bash
# Local unmerged branches with metadata
git branch --no-merged main --format='%(refname:short)|%(committerdate:relative)|%(committerdate:unix)|%(subject)'

# Remote unmerged branches with metadata
git branch -r --no-merged main --format='%(refname:short)|%(committerdate:relative)|%(committerdate:unix)|%(subject)' | grep -v '^origin/main$' | sed 's|^origin/||'

# Commit counts ahead of main for each branch
for branch in $(git branch --no-merged main --format='%(refname:short)'); do
  echo "$branch|$(git rev-list --count main..$branch)"
done

# Active worktrees — maps branch names to worktree paths
git worktree list --porcelain
```

**Worktree detection:** Parse `git worktree list --porcelain` to find which branches have active worktrees. A branch with an active worktree should **never** be defaulted to `delete` — deleting it would break the worktree. Show the worktree path in the branch card and default these to `keep`.

Then cross-reference with GitHub PRs. Use `gh` CLI:

```bash
# For each branch, check PR status
gh pr list --state merged --head "$branch" --json number,title --jq '.[0] // empty'
gh pr list --state closed --head "$branch" --json number,title --jq '.[0] // empty'
gh pr list --state open --head "$branch" --json number,title,url --jq '.[0] // empty'
```

**Auth note:** If the repo uses a non-default GitHub account (check `gh auth status`), switch to the correct account first with `gh auth switch --user <username>`.

### Step 2: Verify delete safety (parallel subagents)

**This is critical.** Before marking any branch as safe to delete, verify it has no unmerged work. Most GitHub repos use **squash merges**, which means:

- Commit hashes on the branch will NEVER appear in main's history
- `git merge-base --is-ancestor` will ALWAYS fail (even for fully merged branches)
- `git log main..branch` will ALWAYS show commits (they're the pre-squash originals)

**These hash-based checks are useless for squash-merge repos.** Use content-based verification instead.

Spawn **parallel subagents** (one per branch, or batched into groups of ~5) to run the verification. Each subagent checks one branch and returns a safety verdict. This is the slowest step — parallelism matters.

#### Verification algorithm for each branch

For branches with a merged PR, run these checks in order:

**Check 0: Squash commit verification (PRIMARY METHOD).** For branches with a merged PR, the most reliable check is to diff the branch tip against the squash merge commit itself:

```bash
# Get the squash merge commit SHA from the PR
merge_sha=$(gh pr view $PR_NUMBER --json mergeCommit --jq '.mergeCommit.oid')

# Diff the branch tip against the squash commit (NOT current main HEAD)
git diff "$merge_sha" "$branch" --stat
```

If empty → the branch tip is byte-for-byte identical to the squash commit on main. **This is definitive proof** the branch has no unmerged work. Mark as `verified-safe` with 0.98 confidence. No further checks needed.

If non-empty → the branch has content that differs from the squash commit. This means either:
- Post-merge commits were added to the branch
- The branch was modified after the PR was squash-merged

In this case, proceed to Check 1 (post-merge commits) and Check 2 (content diff). But for Check 2, diff against the squash commit rather than current main HEAD — this eliminates the massive false-positive diffs from main advancing.

> **Why this is better than diffing against current main HEAD:** Diffing against current main HEAD produces massive false-positive diffs (hundreds of files, thousands of changes) because main has advanced since the branch was merged. These diffs are noise from main's progress, not unique branch work. By diffing against the squash merge commit, we compare the branch to the exact state of main at merge time — any differences are genuinely unique to the branch.

If `gh` is unavailable or the merge commit can't be found, fall back to Checks 1-3 below.

**Check 1: Post-merge commits.** Did anyone commit to the branch AFTER the PR was merged?

```bash
# Get the PR merge timestamp
merged_at=$(gh pr view $PR_NUMBER --json mergedAt --jq '.mergedAt')

# Any commits after that date?
git log "$branch" --after="$merged_at" --oneline
```

If empty → no post-merge work. Move to Check 2.
If non-empty → the branch has new work. Flag it and check if that work exists elsewhere (Check 3).

**Check 2: Content diff.** Compare the branch tip to the squash merge commit (if available from Check 0) or current main tip as a fallback. Use a two-argument diff (not three-dot) to see what's actually different right now:

```bash
# Preferred: diff against squash merge commit (eliminates false positives from main advancing)
git diff "$merge_sha" "$branch" --stat

# Fallback (if merge SHA unavailable): diff against current main
git diff main "$branch" --stat
```

If empty → branch tip is identical to the comparison point. **Confirmed safe to delete.**
If non-empty → the branch has content not in the comparison point. This could be:
  - Pre-squash versions of commits that DID land in main (the content is there, just with different hashes) — check with a targeted file-level comparison
  - Genuinely unmerged work — flag for review

**Check 3: Cross-branch presence.** For post-merge commits or unique content, check if the same work exists on another active branch:

```bash
# Find branches containing this commit (by message, since hashes differ in squash merges)
commit_msg=$(git log -1 --format='%s' "$branch")
git log --all --oneline --grep="$commit_msg"
```

If the commit message appears on another branch (e.g., an open PR branch) with identical content (`git diff <commit_a> <commit_b> --stat` is empty), it's safe — the work lives elsewhere.

#### Safety verdicts

Each subagent returns one of:

| Verdict | Meaning | Dashboard tag |
|---------|---------|---------------|
| `verified-safe` | No post-merge commits, or all content in main/another branch | ✅ Verified safe |
| `likely-safe` | Minor diff (≤3 files, no new files), probably just squash artifacts | ⚠️ Likely safe |
| `has-unique-work` | Branch has content not in main or any other branch | 🔴 Has unique work |
| `needs-review` | Couldn't determine automatically | 🟡 Needs review |

Only branches with `verified-safe` or `likely-safe` get the `delete` default action. Branches with `has-unique-work` default to `next-up` or `park` depending on age. Branches with `needs-review` default to `park`.

### Step 3: Assign default actions

Based on the data and safety verdicts:

- **Active worktree** → `keep` (deleting would break the worktree, regardless of other factors)
- **PR merged + verified-safe** → `delete`
- **PR closed (not merged)** → `delete` (work was intentionally rejected)
- **PR merged + has-unique-work** → `next-up` (unmerged work needs attention)
- **PR open** → `keep`
- **No PR, < 7 days old** → `keep` (likely active work)
- **No PR, 7-30 days old** → `next-up` (worth reviewing)
- **No PR, > 30 days old** → `park` (stale, needs a decision)
- **Release automation branches** (e.g. `release-please-*`) → `keep`

## Page Generation

Build a single-file HTML page using **Vue 3** (CDN global build) following the enhanced playground patterns.

### Required features

1. **Header** with summary stats (total branches, counts per action)

2. **Tab navigation** with these tabs:
   - **All** — every branch, grouped by current action assignment
   - **Work Queue** — only `next-up` branches, drag-to-reorder for prioritization
   - **Delete** — branches marked for deletion, with generated cleanup scripts
   - **Park** — parked/deferred branches
   - **Script** — combined cleanup script for all `delete` branches, with copy buttons

3. **Branch cards** with:
   - Branch name (monospace)
   - Tags: PR status (merged/closed/open/none), local/remote, commit count, age
   - One-line description (last commit subject)
   - Expandable detail section (last commit, age, recommendation)
   - **Action selector** — dropdown or segmented control to change the triage action
   - Checkbox for batch operations

4. **Work Queue tab** features:
   - Drag-to-reorder (or up/down buttons on mobile) to set priority
   - Each item shows why it's worth tackling (commits ahead, description)
   - "Start working" button that copies the `git checkout <branch>` command

5. **Delete tab** features:
   - Separate sections grouped by safety verdict: ✅ Verified safe → ⚠️ Likely safe → 🟡 Needs review → 🔴 Has unique work
   - Each card shows its `safetyDetail` explanation (e.g., "No post-merge commits. Content identical to main.")
   - Generated `git branch -D` and `git push origin --delete` scripts
   - Copy buttons per section and for the full combined script
   - Select all / deselect all per section
   - Only `verified-safe` branches are pre-selected for deletion; others require explicit opt-in

6. **Park tab** features:
   - Parked branches with a note about why they're parked (editable)
   - "Move to Next Up" button to un-park

7. **Persistent state** — all action assignments, queue order, and notes persist to `localStorage` so the user can close the tab and come back

8. **URL state** — active tab syncs to `?tab=` query param via `replaceState`

9. **Mobile-optimized** — touch-friendly tap targets, swipeable tabs, responsive layout

### Design

- Dark theme matching the project aesthetic (GitHub dark palette)
- System font for UI, monospace for branch names and code
- Sticky header with stats
- Fixed bottom action bar when items are selected
- Toast notifications for copy actions

### Data embedding

Embed the branch data as a JavaScript array directly in the HTML file — no external data files. Structure each branch object like:

```javascript
{
  name: 'branch-name',
  prStatus: 'merged' | 'closed' | 'open' | 'no-pr',
  prNumber: 67,          // null if no PR
  prTitle: 'PR title',   // null if no PR
  prUrl: 'https://...',  // null if not open
  local: true,
  remote: true,
  worktree: '/path/to/worktree',  // null if no active worktree
  commits: 14,           // commits ahead of main
  age: '9 weeks',
  ageUnix: 1234567890,   // for sorting
  lastCommit: 'feat: ...',
  description: 'Human-readable summary',
  defaultAction: 'delete',
  safety: 'verified-safe' | 'likely-safe' | 'has-unique-work' | 'needs-review',
  safetyDetail: 'No post-merge commits. Content identical to main.',  // human-readable explanation
  recommendation: 'Safe to delete. PR #67 was merged. Verified: no unique content.'
}
```

## Offline / No-Terminal Workflow (AI prompt + template)

When the user can't run the full skill (e.g. they're on their phone, or in a chat-only AI without terminal), there's a two-step workflow using a bundled script and HTML template.

### Step 1: Gather data

Run the bundled script from the repo root. It needs only `python3`, `git`, and optionally `gh`:

```bash
python3 ${CLAUDE_PLUGIN_ROOT}/skills/branch-triage/scripts/gather_branch_data.py --copy
```

This gathers all branch metadata, PR status (if `gh` is available), diffs, and commit lists. It outputs a prompt to stdout and copies it to clipboard with `--copy`.

The prompt asks the AI to respond with **JSON only** conforming to the dashboard schema — no prose, no HTML, just data.

### Step 2: Paste into any AI, get JSON back

Paste the prompt into ChatGPT, Claude, or any AI. The AI responds with a JSON object. Copy that JSON.

### Step 3: Open the template

Open `${CLAUDE_PLUGIN_ROOT}/skills/branch-triage/template.html` in a browser (double-click or `open` command). Paste the AI's JSON into the textarea, click "Load Dashboard", and the full interactive triage dashboard renders.

### JSON schema

The AI's JSON must conform to this structure:

```json
{
  "repo": "repo-name",
  "branches": [
    {
      "name": "branch-name",
      "prStatus": "merged|closed|open|no-pr",
      "prNumber": 42,
      "local": true,
      "remote": false,
      "worktree": "/path/to/worktree or null",
      "commits": 5,
      "behind": 12,
      "age": "3 days ago",
      "lastCommit": "feat: add example feature",
      "safety": "verified-safe|likely-safe|has-unique-work|needs-review",
      "safetyDetail": "Explanation of safety analysis.",
      "confidence": 0.95,
      "action": "delete|keep|next-up|park|rebase-pr|cherry-pick",
      "recommendation": "One sentence recommendation."
    }
  ]
}
```

The template normalizes field names flexibly — it accepts both `camelCase` and `snake_case` variants (e.g. `prNumber` or `pr_number`, `lastCommit` or `last_commit`). It also persists loaded data to `localStorage`, so the user can close the tab and come back without re-pasting.

## Tips

- The page should look good and be fully functional on first load — sensible defaults everywhere
- Group branches visually by action in the "All" tab so the user gets an instant overview
- Use color coding consistently: green for keep, red for delete, blue for next-up, yellow for park, purple for rebase/cherry-pick
- The generated scripts should be copy-paste ready — no placeholders or TODOs
- Show a count of "X branches to delete, freeing Y local + Z remote refs" in the Delete tab header
